# 后端架构草案

## 目标

构建一个企业级、可扩展、可维护的模块化单体后端，支撑：

- 多模型接入
- 长期记忆
- 用户画像
- 会话检索
- 上下文压缩
- 工具调用
- 异步后台任务

## 架构选择

推荐采用：**模块化单体 + 四层架构 + 按业务模块分包**。

核心原则：

- 顶层按业务模块拆分，而不是全局平铺 `services/ repositories/ models`
- 每个模块内部再按 `api / application / domain / infrastructure` 分层
- 一期保持单体部署，降低复杂度
- 代码边界按未来拆服务的方式预留
- 不采用纯三层硬拼，也不一开始拆微服务

## 分层定义

### 1. `api`

定位：

- 对外 HTTP 接口层

职责：

- FastAPI 路由
- 请求/响应模型
- 参数校验
- 鉴权依赖
- HTTP 异常映射
- 流式输出封装

不负责：

- 复杂业务流程编排
- 直接写 SQL
- 直接调用第三方 SDK

### 2. `application`

定位：

- 用例编排层

职责：

- 单次请求用例编排
- 协调多个模块与仓储接口
- 控制事务边界
- 调用外部能力抽象
- 投递异步任务

不负责：

- 定义核心业务规则
- 直接操作数据库细节
- 处理 HTTP 输入输出

### 3. `domain`

定位：

- 核心业务层

职责：

- 实体
- 值对象
- 领域策略
- 领域服务
- 仓储接口

说明：

- `domain` 放的是业务语义和业务规则
- `domain` 不是 DAO 层
- `domain` 也不是 ORM Model 层
- `domain` 不依赖 FastAPI、SQLAlchemy、Mem0、Honcho、Redis

### 4. `infrastructure`

定位：

- 技术实现层 / 外部依赖接入层

职责：

- Repository 具体实现
- ORM Model
- PostgreSQL / Redis / Queue 接入
- FTS 检索实现
- Mem0 / Honcho 适配器
- LLM Provider 适配器
- 缓存、审计、日志落地

不负责：

- 编排完整业务流程
- 定义业务规则

### 5. `shared`

定位：

- 跨模块共享的基础能力层

职责：

- 配置
- 统一异常
- 基础类型
- 通用工具函数
- 基础内核抽象

注意：

- 只放真正通用的内容
- 不要把某个业务模块的逻辑塞进 `shared`
- 不要让 `shared` 变成大杂烩

## 模块划分

### 核心模块

- `chat`：聊天回合、消息生命周期、流式输出
- `agent`：agent loop、prompt 组装、召回、压缩、工具循环、provider 路由
- `memory`：内建长期记忆、写回、去重、Mem0 同步
- `profile`：用户画像、画像合并、Honcho 同步
- `session`：会话元数据、消息归档、FTS 检索、摘要来源
- `tools`：工具定义、权限校验、审批、调用日志
- `models`：provider 管理、模型能力、路由和 fallback

### 支撑模块

- `admin`：健康检查、审计、任务管理、治理接口
- `integrations`：跨模块共享的外部技术集成
- `jobs`：异步任务入口

## 全局 `api` 与模块 `api` 的区别

### 全局 `app/api/`

定位：

- 整个后端应用共享的 API 装配层

负责：

- 聚合所有模块路由
- 提供全局依赖注入入口
- 放全局中间件
- 统一异常映射
- 定义通用响应结构和分页模型

适合放：

- `app/api/router.py`
- `app/api/deps.py`
- `app/api/errors.py`
- `app/api/middleware/`
- `app/api/schemas/common.py`

### 模块 `modules/*/api/`

定位：

- 某个业务模块自己的接口适配层

负责：

- 定义该模块自己的路由
- 定义该模块自己的请求响应模型
- 把 HTTP 请求转成该模块的 use case 调用
- 输出该模块自己的 presenter/view model

适合放：

- `modules/chat/api/router.py`
- `modules/memory/api/schemas.py`
- `modules/tools/api/presenters.py`

### 一句话区分

- 全局 `api` 解决“整个应用对外怎么统一暴露”
- 模块 `api` 解决“某个业务模块的接口怎么定义和组织”

### 类比

- 全局 `api` 像总路由中心和公共网关层
- 模块 `api` 像每个业务域自己的 controller 包

## 层职责落地说明

### `api` 层存放什么

建议放：

- `router.py`
- `schemas.py`
- `presenters.py`
- 依赖注入入口

典型内容：

- `ChatController`
- `MemoryController`
- `ProfileController`

### `application` 层存放什么

建议放：

- `use_cases/`
- `dto.py`
- `commands.py`
- `queries.py`
- 应用服务
- orchestrator

典型内容：

- `SendMessageUseCase`
- `UpdateMemoryUseCase`
- `SearchSessionUseCase`
- `DispatchToolUseCase`

### `domain` 层存放什么

建议放：

- `entities/`
- `value_objects/`
- `policies/`
- `services/`
- `repositories.py`

典型内容：

- `Session`
- `Message`
- `MemoryItem`
- `UserProfile`
- `ModelRoute`
- `ToolInvocation`

典型规则：

- 什么信息可以写成长期记忆
- 哪些工具需要审批
- 何时需要召回历史上下文
- 何时触发模型 fallback

### `infrastructure` 层存放什么

建议放：

- `orm/`
- `repositories/`
- `providers/`
- `search/`
- `registry/`
- `executors/`

典型内容：

- `SqlSessionRepository`
- `SqlMemoryRepository`
- `Mem0Adapter`
- `HonchoAdapter`
- `OpenAIProvider`
- `AnthropicProvider`
- `PostgresFtsSessionSearch`

### `shared` 层存放什么

建议放：

- `config`
- `exceptions`
- 基础仓储抽象
- 基础实体抽象
- 通用类型
- 通用工具函数

不建议放：

- 具体业务模块的 helper
- 与某个模块强绑定的规则
- 第三方 SDK 调用

## Application 层核心用例

### `SendMessageUseCase`

职责：

- 接收用户输入
- 拉取必要上下文
- 组装 prompt
- 调用主模型
- 处理 tool calls
- 持久化消息与会话
- 投递异步任务

### `SearchSessionUseCase`

职责：

- FTS 检索历史会话
- 提取相邻上下文
- 调用摘要模型整合结果
- 返回给当前回合

### `UpdateMemoryUseCase`

职责：

- 写入内建记忆
- 执行去重、替换、删除策略
- 调用 Mem0 同步
- 记录来源与版本

### `UpdateProfileUseCase`

职责：

- 从交互中抽取画像信号
- 更新内部画像
- 执行画像合并策略
- 调用 Honcho 同步

### `CompressContextUseCase`

职责：

- 估算 token
- 裁剪旧 tool outputs
- 保留 head / middle / tail
- 生成结构化摘要
- 支持迭代更新

## Agent 模块设计重点

`agent` 是系统核心，不建议做成单个 `service.py`。

建议拆分为：

- `AgentLoopOrchestrator`
- `PromptAssembler`
- `RecallOrchestrator`
- `MemoryOrchestrator`
- `ContextEngine`
- `ToolDispatchOrchestrator`
- `ProviderRouteOrchestrator`

这样可以把 prompt 组装、召回、压缩、工具调度、模型路由分开治理，避免 `application` 层出现超大上帝类。

## 内部服务，不对外暴露 API

以下能力一期保留为后端内部服务，不直接对前端开放接口：

### `memory service`

内部职责：

- 写入内建记忆
- 替换旧记忆
- 删除无效记忆
- 搜索长期记忆
- 同步 Mem0

原因：

- 这属于 Agent Runtime 的内部能力
- 一期不做用户手动管理记忆页面
- 直接暴露 CRUD 会增加接口复杂度与安全边界成本

### `tool service`

内部职责：

- 工具注册
- 工具调度执行
- 工具审批流程
- 工具调用日志

原因：

- 工具调用属于 Agent Loop 的内部执行链路
- 前端只需要在聊天过程中承接审批交互，不需要直接拥有完整工具控制 API
- 工具日志一期作为内部调试与治理能力保留

### `session search service`

内部职责：

- FTS 检索历史会话
- 提取相邻上下文
- 调用摘要模型生成 recall summary
- 回注当前对话链路

原因：

- 一期重点是服务当前轮对话，不是给前端单独提供 session 内搜索产品功能
- 该能力更适合作为 Recall Orchestrator 的内部组成部分

### 其他内部能力

- `recall service`
- `compression service`
- `provider router`
- `agent loop orchestrator`

这些都属于系统运行时内部能力，不直接暴露为对外 API。

## 运行原则

- `api` 只负责 HTTP 输入输出
- `application` 只做编排，不写 SQL
- `domain` 只关心业务语义和规则
- `infrastructure` 只实现接口，不定义业务规则
- `shared` 只放真正通用的基础能力
- ORM Model 只放在 `infrastructure/orm`
- Repository 接口定义在 `domain`
- Repository 实现在 `infrastructure/repositories`
- 未来拆服务时按模块平滑拆分

## 推荐落地顺序

1. 先搭好 `api / core / shared / modules / integrations / jobs`
2. 先实现 `chat / agent / memory / profile / session / tools / models`
3. 先落 PostgreSQL + FTS 检索
4. 再接入 Mem0 和 Honcho
5. 再补 Redis 队列与异步任务
6. 最后做企业级权限、审计、多租户

## 结论

这套结构适合当前项目的原因是：

- 能承载长期记忆、画像演化和 agent runtime 的复杂性
- 能把检索、压缩、路由、工具调用拆开治理
- 模块边界清晰，维护性高
- 后续扩展新 provider、工具和后台任务的代价低
- 未来可以从模块化单体平滑演进到服务拆分
