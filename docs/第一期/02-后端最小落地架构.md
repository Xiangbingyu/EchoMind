# 后端最小落地架构

## 目标

在不牺牲后续扩展能力的前提下，把后端结构压缩到可以直接开工的一期版本。

适用范围：

- 聊天主链路
- Agent Loop
- 长期记忆
- 用户画像
- Session Search
- 模型路由
- 工具调用

不追求一步到位的完整企业级目录，而是先做**最小完备、可演进**的模块化单体。

## 一期原则

- 保留分层思想，但减少目录深度。
- 保留模块边界，并让核心能力可以独立封装、独立迁移。
- 先做模块化单体，不做微服务。
- 先做核心主链路，后台任务和治理能力做最小实现。
- 内建能力作为系统真源，外接能力作为增强层与适配层。
- 优先保证可读性、可维护性、可交付。
- 优先按业务能力组织目录，避免把核心能力打散在横向层目录里。

## 一期推荐架构

推荐采用：**模块化单体 + 按能力封装的可迁移模块 + 模块内轻量分层**。

一期不再以全局 `api/services/domain/infra` 横向切目录为主，而是以核心能力模块为主组织代码：

- `chat`：对话接入与流式输出
- `agent`：主链路编排与 runtime
- `memory`：长期记忆真源与同步
- `profile`：用户画像真源与同步
- `session`：会话归档与检索
- `models`：模型路由与 provider 管理
- `tools`：工具注册、审批、执行、审计
- `shared`：跨模块真正通用的基础能力
- `platform`：数据库、队列、缓存等底座能力

说明：

- 每个核心模块都尽量做到高内聚、低耦合。
- 模块内部可以保留轻量的 `api/service/domain/infra` 思想，但不要求目录过深。
- 模块之间通过明确的方法边界、repository、provider、facade 交互，而不是互相直接穿透内部实现。
- 一期先以单体部署，二期若需要拆服务，可按模块边界直接迁移。

## 模块封装原则

- 每个核心模块都应有清晰的职责闭环，而不是只承担某一层代码。
- 模块应暴露有限的公开接口，优先通过 `service` / `facade` 被其他模块调用。
- 模块内部持有自己的 `domain`、`repository`、`provider adapter`，避免公共目录承载过多业务语义。
- 跨模块共享的内容只进入 `shared`，禁止把业务逻辑沉淀到 `shared`。
- 如果未来要拆服务，优先迁移整个模块，而不是从横向层里拆碎片代码。

- `profile` 将来可以独立迁移为用户画像服务或用户服务的一部分。
- `tools` 将来可以独立迁移为工具执行服务。
- `agent` 将来可以独立迁移为主链路编排服务。
- `session` 将来可以独立迁移为会话与检索服务。

## 模块内职责

一期推荐“模块为主，层次为辅”。

也就是说，不是全局统一放一个 `services/` 或 `domain/`，而是在每个模块内部保留最小必要分层：

### 1. `api.py`

职责：

- 暴露该模块对 HTTP 的入口
- 请求参数校验
- 响应模型
- 流式输出适配

### 2. `service.py` / `orchestrator.py`

职责：

- 模块公开能力入口
- 业务编排
- 对外提供稳定调用方法
- 串联 repository、provider、其他模块 facade

说明：

- 简单模块可用 `service.py`
- 对 `agent` 这类强编排模块，可显式使用 `orchestrator.py`

### 3. `domain.py`

职责：

- 核心实体
- 值对象
- 少量关键规则

一期不要把 `domain` 做得太重，先保留最关键的业务语义。

### 4. `repository.py` / `provider.py`

职责：

- 模块自己的数据访问边界
- 外部能力适配边界
- 避免其他模块直接依赖数据库细节或第三方 SDK

### 5. `shared` 与 `platform`

职责：

- `shared` 放真正跨模块复用的异常、日志、schema、工具函数
- `platform` 放数据库、缓存、队列等底座实现

原则：

- 业务逻辑尽量留在模块内
- 横向公共目录不承载具体业务语义

## 一期核心模块

### 1. `chat`

负责：

- 接收用户消息
- 流式输出
- retry / regenerate / stop

### 2. `agent`

负责：

- 主循环
- prompt 组装
- 调用 `tools` 模块执行 function calling
- tool result 回注
- fallback

建议进一步拆成两个内部单元：

- `agent/orchestrator`：负责主链路编排、状态推进、回退策略
- `tools/service|dispatcher`：负责工具注册、审批、执行、日志

### 3. `memory`

负责：

- 内建轻量记忆真源
- 记忆抽取、去重、合并、替换
- 长期记忆召回
- 外部记忆 provider 同步
- 外部记忆 provider 适配与切换

说明：

- 内建记忆是系统真源
- 外接长期记忆系统属于增强层，不作为主数据源
- 一期先实现本地真源 + 一个外部 provider（如 Mem0）
- 设计上应保证后续可整体迁移成独立 `memory service`

### 4. `profile`

负责：

- 内建用户画像真源
- 画像抽取、合并、刷新、版本化
- 外部画像 provider 同步
- 外部画像 provider 适配与切换

说明：

- 内建画像是系统真源
- 外接画像系统属于增强层，不作为主数据源
- 一期先实现本地真源 + 一个外部 provider（如 Honcho）
- 设计上应保证后续可整体迁移成独立 `profile service` 或并入 `user service`

### 5. `session`

负责：

- 会话归档
- 历史消息读取
- FTS 检索
- session summary 来源

### 6. `models`

负责：

- provider 配置
- 模型切换
- fallback 模型

### 7. `tools`

负责：

- 工具注册
- 工具调用
- 工具审批
- 工具日志

说明：

- 一期先作为单体内独立模块存在
- 与 `agent` 通过明确接口交互，不与主链路代码混写
- 后续可单独迁移成 `tool service`

- `SessionRepository`
- `MessageRepository`
- `MemoryRepository`

说明：

- repository 优先定义在各自模块内部
- 只有跨模块需要时，才暴露最小查询接口

### Provider / Adapter

- `LLMProvider`

- `Mem0Adapter`：`MemoryProvider` 的一个实现
- `HonchoAdapter`：`ProfileProvider` 的一个实现

### Service

- `ChatService`
- `AgentOrchestrator`
- `MemoryService`
- `ProfileService`
- `SessionService`
- `ModelService`
- `ToolService`
- `PromptAssembler`
- `RecallService`
- `ToolDispatcher`

说明：

- 一期不必全部先抽成复杂 interface

- 这些 service 应优先作为模块公开入口，而不是散落在全局目录

## 一期 FastAPI 路由

说明：

- 这里列的是**一期对前端或外部调用方暴露的 HTTP API**。
- 前文提到的模块公开入口，优先指模块内的 `service` / `facade` / `orchestrator` 方法边界，不等同于一期必须全部暴露为 REST 接口。
- 是否暴露 HTTP API，取决于该能力是否需要被前端直接调用、被后台运营调用，或被外部系统独立集成。

## 一期内部服务，不对外暴露 API

以下能力一期保留为后端内部服务：

- `MemoryService`、`ProfileService`、`ToolService`、`RecallService`、`CompressionService` 的“公开”主要是指模块边界清晰、可被其他模块调用，不代表一期要直接开放成 HTTP API。
- 这样做的目的，是先把能力封装成可迁移模块，再根据真实调用场景决定是否升级为独立服务接口。
- 如果二期需要拆服务，这些模块可以各自补充对外 API，而不必先在一期为了“可拆”而把所有内部能力都暴露出去。

### `MemoryService`

负责：

- 写入内建记忆
- 替换记忆
- 删除记忆
- 召回记忆
- 同步外部记忆 provider

说明：

- 一期先不做用户手动管理记忆页面
- 记忆维护由 Agent Runtime 内部自动完成
- 如果二期需要提供记忆运营台、人工编辑记忆、跨应用共享记忆，再补充独立 `memory API`

### `ProfileService`

负责：

- 刷新内建画像
- 合并画像变更
- 生成画像版本
- 同步外部画像 provider

说明：

- 一期画像维护以系统内部真源为主
- 外部画像平台用于增强与同步，不作为主链路硬依赖
- 如果后续需要账号中心、用户偏好设置页、跨端资料同步，再补充独立 `profile/user API`

### `ToolService`

负责：

- 工具注册
- 工具执行
- 工具审批
- 工具日志

说明：

- 工具链路属于 Agent Loop 内部执行过程
- 前端只需要在聊天过程中承接必要审批，不直接控制完整工具 API
- 如果后续要做统一工具平台、工具审计后台、第三方工具接入，再补充独立 `tool API`

### `RecallService`

负责：

- 历史 session 检索
- 提取相邻上下文
- 召回结果摘要
- 回注当前对话链路

说明：

- 一期不单独暴露 session 内搜索接口
- recall 作为主聊天链路的内部能力存在
- 如果后续要支持全局搜索、知识运营检索，再补充独立 `search/recall API`

### `CompressionService`

负责：

- token 估算
- 旧 tool outputs 裁剪
- 中段对话压缩
- 结构化摘要生成

说明：

- 压缩属于运行时治理能力，不直接暴露为业务 API

### 其他内部能力

- `AgentService`
- `PromptAssembler`
- `ToolDispatcher`
- `ContextCompressor`
- `ProviderRouter`

这些都属于后端内部运行时能力，不直接对前端开放。

## 一期同步主链路

同步处理：

1. 接收用户消息
2. 加载 session
3. 组装 prompt
4. 执行必要 recall
5. 判断是否需要压缩
6. 调用主模型
7. 执行 tool call
8. 回写消息
9. 返回响应

## 一期异步任务

先只做最有价值的异步项：

- memory writeback to external provider
- profile sync to external provider
- session indexing

可以先用：

- FastAPI BackgroundTasks 或
- Redis + Arq

不要一期就把任务系统做得太重。

## 一期最小数据库建议

优先建议：

- PostgreSQL
- FTS 搜索
- Redis 做缓存和任务队列

如果需要更轻，也可以：

- SQLite 做开发环境
- PostgreSQL 做正式环境

## 一期暂时不要做的事

- 不拆微服务
- 不做过多 domain event
- 不做过多 interface 抽象
- 不做全套 admin 治理后台
- 不做复杂多级 jobs 子系统
- 不做过细的 presenters / commands / queries 目录

## 演进路径

### 一期

- 单体部署
- 按能力模块封装
- 模块内轻量分层
- service / orchestrator 作为模块公开入口
- domain 保持轻量

### 二期

- 模块内部拆成更清晰四层
- 仓储接口与 provider 接口正式抽象
- 高频模块独立部署为单独服务
- jobs / admin / integrations 细化
- 多租户、审计、治理增强

### 三期

- 如果业务复杂度足够高，再按模块拆服务
- 沿用一期确定的模块边界，避免重新切分主链路

## 最终建议

你现在最合适的不是“最优雅的终局架构”，而是：

- 一套能支撑核心业务
- 目录不复杂到影响开发速度
- 还能平滑演进

所以一期最小落地架构建议就是：

- 用模块化单体
- 优先按 `chat / agent / memory / profile / session / models / tools` 封装
- 在模块内部保留轻量分层，而不是全局横向分层
- 把复杂度集中在 `agent/orchestrator` 与各模块公开 service 上
- 从一期开始就为后续拆成独立服务预留迁移边界
