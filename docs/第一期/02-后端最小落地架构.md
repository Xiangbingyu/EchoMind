# 后端最小落地架构

## 目标

在不牺牲后续扩展能力的前提下，把后端结构压缩到可以直接开工的一期版本。

适用范围：

- 聊天主链路
- 用户登录与身份认证
- Agent Loop
- 长期记忆
- 用户画像
- Session Search
- 模型路由
- 工具调用

不追求一步到位的完整企业级目录，而是先做**最小完备、可演进**的模块化单体。

## 一期原则

- 保留分层思想，但减少目录深度。
- 保留模块边界，并让核心能力可以独立封装、独立迁移。
- 先做模块化单体，不做微服务。
- 先做核心主链路，后台任务和治理能力做最小实现。
- 内建能力作为系统真源，外接能力作为增强层与适配层。
- 优先保证可读性、可维护性、可交付。
- 优先按业务能力组织目录，避免把核心能力打散在横向层目录里。

## 一期推荐架构

推荐采用：**模块化单体 + 按能力封装的可迁移模块 + 模块内轻量分层**。

一期不再以全局 `api/services/domain/infra` 横向切目录为主，而是以核心能力模块为主组织代码：

- `chat`：对话接入与流式输出
- `user`：用户账号、登录认证与身份上下文
- `agent`：主链路编排与 runtime
- `memory`：长期记忆真源与同步
- `profile`：用户画像真源与同步
- `session`：会话归档与检索
- `models`：模型路由与 provider 管理
- `tools`：工具注册、自动执行、日志与审计
- `shared`：跨模块真正通用的基础能力
- `platform`：数据库、队列、缓存等底座能力

说明：

- 每个核心模块都尽量做到高内聚、低耦合。
- 模块内部可以保留轻量的 `api/service/domain/infra` 思想，但不要求目录过深。
- 模块之间通过明确的方法边界、repository、provider、facade 交互，而不是互相直接穿透内部实现。
- 一期先以单体部署，二期若需要拆服务，可按模块边界直接迁移。

## 模块封装原则

- 每个核心模块都应有清晰的职责闭环，而不是只承担某一层代码。
- 模块应暴露有限的公开接口，优先通过 `service` / `facade` 被其他模块调用。
- 模块内部持有自己的 `domain`、`repository`、`provider adapter`，避免公共目录承载过多业务语义。
- 跨模块共享的内容只进入 `shared`，禁止把业务逻辑沉淀到 `shared`。
- 如果未来要拆服务，优先迁移整个模块，而不是从横向层里拆碎片代码。

## 模块内职责

一期推荐“模块为主，层次为辅”。

也就是说，不是全局统一放一个 `services/` 或 `domain/`，而是在每个模块内部保留最小必要分层：

### 1. `api.py`

职责：

- 暴露该模块对 HTTP 的入口
- 请求参数校验
- 响应模型
- 流式输出适配

### 2. `service.py` / `orchestrator.py`

职责：

- 模块公开能力入口
- 业务编排
- 对外提供稳定调用方法
- 串联 repository、provider、其他模块 facade

说明：

- 简单模块可用 `service.py`
- 对 `agent` 这类强编排模块，可显式使用 `orchestrator.py`

### 3. `domain.py`

职责：

- 核心实体
- 值对象
- 少量关键规则

一期不要把 `domain` 做得太重，先保留最关键的业务语义。

### 4. `repository.py` / `provider.py`

职责：

- 模块自己的数据访问边界
- 外部能力适配边界
- 避免其他模块直接依赖数据库细节或第三方 SDK

### 5. `shared` 与 `platform`

职责：

- `shared` 放真正跨模块复用的异常、日志、schema、工具函数
- `platform` 放数据库、缓存、队列等底座实现

原则：

- 业务逻辑尽量留在模块内
- 横向公共目录不承载具体业务语义

## 一期核心模块

### 1. `user`

负责：

- 账号密码登录
- JWT access token 签发与校验
- 通过 Cookie 传递 token
- 从 token 中解析当前 `user_id`
- 使用 Redis 管理 token 状态
- 同域部署下的认证与 CSRF 约束

说明：

- 一期只做最小认证闭环，不做 refresh token。
- 一期默认所有业务链路都绑定 `user_id`，不再采用“伪单用户上下文”。
- `profile` 关注用户画像真源与用户名称，`user` 只关注身份、登录态与账号凭证。
- 一期按同域部署设计，access token 通过 `HttpOnly Cookie` 传递。
- JWT claims 至少包括：`sub(user_id)`、`jti`、`exp`。
- Redis 以 `auth:token:{jti}` 维护 token 状态，支持登出失效与用户状态校验。
- 为避免 Cookie 登录态的 CSRF 风险，一期至少应校验 `Origin/Referer`，如后续跨站再升级为专门的 CSRF token 方案。
- 一期首个可登录账号通过 seed 脚本初始化，不单独开放注册接口。

### 2. `chat`

负责：

- 接收用户消息
- 流式输出
- retry / regenerate / stop
- 管理 chat run 生命周期与运行态

说明：

- `chat` 负责对 HTTP / SSE 的接入与流式适配，不拥有 `sessions/messages/session_summaries` 的写表边界。
- 一期引入 `run_id` 作为一次对话运行的标识，用于串联流式输出、停止、重试与重新生成。
- `POST /stop` 采用软取消语义：将当前 run 标记为 `cancel_requested`，由 `agent runtime` 在安全点停止模型流、工具调用和后续写回。
- 一期 chat 运行态建议放 Redis，例如用 `chat:run:{run_id}` 记录 `status`、`session_id`、`user_id`、`cancel_requested` 等短生命周期控制字段。
- `retry` 不创建子 session，而是在原 session 内重新执行当前重试目标。
- `regenerate` 创建新的分支 session，用于保留多种候选回复结果并支持切换。
- `regenerate` 产生的分支不出现在 session 列表中，而是在对应对话点以下“候选结果”方式切换查看。
- 当用户基于某个 `regenerate` 分支继续对话时，应先将该分支收敛为当前主分支；一旦继续对话，该对话点不再允许切换主分支，只保留历史候选结果查看能力。
- 同一分支点下“唯一主结果”的更细约束，一期留到 `session` 模块详细设计阶段收口。

### 3. `agent`

负责：

- 主循环
- prompt 组装
- 调用 `tools` 模块执行 function calling
- tool result 回注

建议进一步拆成两个内部单元：

- `agent/orchestrator`：负责主链路编排、状态推进、回退策略
- `tools/service|dispatcher`：负责工具注册、自动执行、日志

### 4. `memory`

负责：

- 内建轻量记忆真源
- 记忆抽取、去重、合并、替换
- 长期记忆召回
- 外部记忆 provider 同步
- 外部记忆 provider 适配与切换

说明：

- 内建记忆是系统真源
- 外接长期记忆系统属于增强层，不作为主数据源
- 一期先实现本地真源 + 一个外部 provider（如 Mem0）
- 同一 `(user_id, scope, content_hash)` 的记忆生命周期内复用原记录，通过 `state/version` 表达替换、删除和恢复，而不是重复插入新行。
- 设计上应保证后续可整体迁移成独立 `memory service`

### 5. `profile`

负责：

- 内建用户画像真源
- 画像抽取、合并、刷新、版本化
- 外部画像 provider 同步
- 外部画像 provider 适配与切换

说明：

- 内建画像是系统真源
- 外接画像系统属于增强层，不作为主数据源
- 一期先实现本地真源 + 一个外部 provider（如 Honcho）
- 设计上应保证后续可整体迁移成独立 `profile service` 或并入 `user service`

### 6. `session`

负责：

- 会话归档
- 历史消息读取
- FTS 检索
- session summary 来源
- 分支会话元数据管理

说明：

- `session` 是 `sessions`、`messages`、`session_summaries` 的 owner。
- `chat` 通过 `SessionService` 创建与回写消息，不直接跨模块拼 SQL。
- `RecallService` 与 `CompressionService` 读取或写入摘要时，也应通过 `session` 模块边界进入。
- `regenerate` 的分支可见性、候选结果切换、主分支锁定状态由 `session` 模块负责解释和聚合。
- `branch_point_message_id` 应在数据库层指向 `messages.id`，并由应用层额外保证它必须属于 `parent_session_id`。
- `message_count`、`tool_call_count`、`input_tokens`、`output_tokens`、`last_message_at` 等聚合字段由 `SessionService` 单点维护，并提供离线重建能力作为兜底。

### 7. `models`

负责：

- provider 配置
- 用户级模型切换
- 默认模型解析

说明：

- 一期模型路由按 `user_id` 生效，不做全局共享切换。
- `/api/v1/models/*` 相关接口都基于当前登录用户上下文读取和修改模型路由。
- 一期不设计系统级 fallback 模型，只保留用户主模型选择与平台默认模型回退。
- 平台默认模型在一期按全局唯一约束设计。
- `model_routes` 继续保存 `provider + model` 文本组合，并由应用层校验其必须存在于平台模型池且处于启用状态。
- 模型禁用、悬空引用和更细的路由闭环规则，留到 `models` 模块详细设计阶段收口。

### 8. `tools`

负责：

- 工具注册
- 工具调用
- 工具日志

说明：

- 一期先作为单体内独立模块存在
- 与 `agent` 通过明确接口交互，不与主链路代码混写
- 一期工具默认自动执行，不引入人工审批流
- 工具调用日志建议记录 `run_id`，用于稳定归属到某次生成尝试
- `tool_invocations` 的去重、`tool_call_id` 关联和查询细则，留到 `tools` 模块详细设计阶段收口。
- 后续可单独迁移成 `tool service`

## 一期 FastAPI 路由

说明：

- 这里列的是**一期对前端或外部调用方暴露的 HTTP API**。
- 前文提到的模块公开入口，优先指模块内的 `service` / `facade` / `orchestrator` 方法边界，不等同于一期必须全部暴露为 REST 接口。
- 是否暴露 HTTP API，取决于该能力是否需要被前端直接调用、被后台运营调用，或被外部系统独立集成。
- 一期仍以聊天主链路为中心，优先暴露用户看得见、前端必须调用的接口。

### `/api/v1/chat`

- `POST /send`
- `POST /stream`
- `POST /stop`
- `POST /retry`
- `POST /regenerate`

### `/api/v1/auth`

- `POST /login`
- `POST /logout`
- `GET /me`

说明：

- `login` 成功后由后端设置 access token Cookie，而不是要求前端手动存储 token。
- `logout` 应同时清理 Cookie 并将当前 `jti` 标记为失效。
- `me` 仅返回登录身份字段，例如 `user_id`、`username`、`status`，不聚合 `profile` 信息。
- 前端若需要用户名称等展示信息，应单独调用 `/api/v1/profile`。

### `/api/v1/sessions`

- `GET /`
- `GET /{session_id}`
- `GET /{session_id}/messages`
- `POST /{session_id}/rename`

说明：

- session 列表只返回主分支会话，不返回 `regenerate` 产生的候选子分支。
- session 详情或消息读取结果应能够携带某个对话点下的候选分支结果，用于前端在消息内切换查看。
- 一期不提供 session 删除能力，避免对对话主骨架和记忆来源指标做破坏性处理。

### `/api/v1/profile`

- `GET /`
- `POST /refresh`

说明：

- 一期不开放人工编辑画像接口，画像只作为系统自动抽取与刷新结果对外只读展示。

- `POST /refresh` 建议走异步刷新流程；刷新失败不影响聊天主链路可用性。
- 一期内部画像刷新任务不单独持久化，只记录刷新后的画像版本、外部同步任务状态和失败审计。

## 一期内部服务，不对外暴露 API

以下能力一期保留为后端内部服务：

- `MemoryService`、`ProfileService`、`ToolService`、`RecallService`、`CompressionService` 的“公开”主要是指模块边界清晰、可被其他模块调用，不代表一期要直接开放成 HTTP API。
- 这样做的目的，是先把能力封装成可迁移模块，再根据真实调用场景决定是否升级为独立服务接口。
- 如果二期需要拆服务，这些模块可以各自补充对外 API，而不必先在一期为了“可拆”而把所有内部能力都暴露出去。

### `MemoryService`

负责：

- 写入内建记忆
- 替换记忆
- 删除记忆
- 召回记忆
- 同步外部记忆 provider

说明：

- 一期先不做用户手动管理记忆页面
- 记忆维护由 Agent Runtime 内部自动完成
- 如果二期需要提供记忆运营台、人工编辑记忆、跨应用共享记忆，再补充独立 `memory API`

### `ProfileService`

负责：

- 刷新内建画像
- 合并画像变更
- 生成画像版本
- 同步外部画像 provider

说明：

- 一期画像维护以系统内部真源为主
- 外部画像平台用于增强与同步，不作为主链路硬依赖
- 聊天主链路不依赖画像刷新成功；画像刷新失败应进入异步重试或错误记录，而不是阻塞回复。
- 一期内部 refresh 流程本身不额外建任务表，优先保持轻量。
- 如果后续需要账号中心、用户偏好设置页、跨端资料同步，再补充独立 `profile/user API`

### `ToolService`

负责：

- 工具注册
- 工具执行
- 工具日志

说明：

- 工具链路属于 Agent Loop 内部执行过程
- 一期工具默认自动执行，前端不参与审批链路
- 如果后续要做统一工具平台、工具审计后台、第三方工具接入，再补充独立 `tool API`

### `RecallService`

负责：

- 历史 session 检索
- 提取相邻上下文
- 召回结果摘要
- 回注当前对话链路

说明：

- 一期不单独暴露 session 内搜索接口
- recall 作为主聊天链路的内部能力存在
- 如果后续要支持全局搜索、知识运营检索，再补充独立 `search/recall API`

### `CompressionService`

负责：

- token 估算
- 旧 tool outputs 裁剪
- 中段对话压缩
- 结构化摘要生成

说明：

- 压缩属于运行时治理能力，不直接暴露为业务 API

### 其他内部能力

- `AgentService`
- `PromptAssembler`
- `ToolDispatcher`
- `ChatRunStateStore`
- `ProviderRouter`

这些都属于后端内部运行时能力，不直接对前端开放。

## 一期同步主链路

同步处理：

1. 校验登录态并解析 `user_id`
2. 创建 `run_id` 并初始化 Redis 运行态
3. 接收用户消息
4. 加载 session
5. 组装 prompt
6. 执行必要 recall
7. 判断是否需要压缩
8. 调用主模型
9. 执行 tool call
10. 回写消息与工具日志
11. 更新 Redis run 状态并返回响应

## 一期异步任务

先只做最有价值的异步项：

- memory writeback to external provider
- profile sync to external provider

可以先用：

- FastAPI BackgroundTasks 或
- Redis + Arq

不要一期就把任务系统做得太重。

补充约束：

- 一期异步任务以 `memory_sync_jobs / profile_sync_jobs` 为主，不默认启用通用 outbox。
- `app_outbox_events` 如需引入，放到二期作为统一事件机制增强项。
- `chat:run:{run_id}` 的 TTL、完成后保留时长和异常清理细则，留到 `chat` 模块详细设计阶段收口。

## 一期最小审计建议

建议至少记录以下事件到 `audit_logs`：

- 登录成功与登录失败
- 登出
- session 重命名
- 模型切换
- 工具执行失败
- profile 刷新失败
- memory/profile 外部 provider 同步进入 dead-letter 或最终失败

## 一期最小数据库建议

优先建议：

- PostgreSQL
- FTS 搜索
- Redis 做缓存、任务队列和 token 状态管理

如果需要更轻，也可以：

- SQLite 做开发环境
- PostgreSQL 做正式环境

补充约束：

- access token 与 chat run 运行态建议统一放 Redis。
- 数据库负责业务真源；Redis 负责短生命周期状态与运行态控制。

## 一期暂时不要做的事

- 不拆微服务
- 不做过多 domain event
- 不做过多 interface 抽象
- 不做全套 admin 治理后台
- 不做复杂多级 jobs 子系统
- 不做 refresh token
- 不开放人工画像编辑接口
- 不做过细的 presenters / commands / queries 目录

## 演进路径

### 一期

- 单体部署
- 按能力模块封装
- 模块内轻量分层
- service / orchestrator 作为模块公开入口
- domain 保持轻量

### 二期

- 模块内部拆成更清晰四层
- 仓储接口与 provider 接口正式抽象
- 高频模块独立部署为单独服务
- jobs / admin / integrations 细化
- 多租户、审计、治理增强

### 三期

- 如果业务复杂度足够高，再按模块拆服务
- 沿用一期确定的模块边界，避免重新切分主链路

## 最终建议

你现在最合适的不是“最优雅的终局架构”，而是：

- 一套能支撑核心业务
- 目录不复杂到影响开发速度
- 还能平滑演进

所以一期最小落地架构建议就是：

- 用模块化单体
- 优先按 `user / chat / agent / memory / profile / session / models / tools` 封装
- 在模块内部保留轻量分层，而不是全局横向分层
- 把复杂度集中在 `agent/orchestrator` 与各模块公开 service 上
- 从一期开始就为后续拆成独立服务预留迁移边界
