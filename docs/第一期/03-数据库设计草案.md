# 数据库设计草案

## 目标

为一期后端提供一套**可直接落地、能支撑 MVP** 的数据库设计。

这一版重点解决：

- 聊天会话与消息存储
- 长期记忆内建存储
- 用户画像内建存储
- 外部 memory / profile provider 同步状态管理
- 会话检索与 FTS
- 工具调用日志
- 审计日志
- 模型配置与当前路由状态

## 设计原则

### 1. 一期只做单用户

一期数据库不考虑多用户、多租户场景，因此不添加这类预留字段。

### 2. 内部数据库是真源，外部平台是同步目标

对于长期记忆和用户画像，一期建议采用：

- 内部数据库作为**系统真源**
- Mem0 / Honcho 作为**外部同步目标**

不要把外部平台当作主数据源。

### 3. 结构化主字段入列，灵活数据放 `jsonb`

建议采用：

- 高频筛选、排序、关联字段放独立列
- 可变结构内容放 `jsonb`

例如：

- `messages.role`、`messages.session_id` 放独立列
- `messages.tool_calls`、`messages.reasoning_details` 放 `jsonb`
- `profiles.preferences`、`profiles.work_habits` 放 `jsonb`

### 4. 一期优先 PostgreSQL

参考 Hermes 的设计，它用 SQLite + WAL + FTS5 完成单机持久化，这对于 CLI 型产品很合适。

但 EchoMind 一期是后端服务形态，建议直接采用：

- PostgreSQL 作为主库
- PostgreSQL FTS 作为会话检索能力
- Redis 作为缓存和轻量任务队列

### 5. 核心主表保持 provider 无关

对于记忆和画像模块，建议采用：

- 核心业务字段只描述系统内部语义
- 不在核心主表中引入过多第三方平台特有字段
- 第三方平台相关信息放到 binding 表或同步任务表

### 6. 外部平台接入采用 binding 表 + 同步任务

建议采用：

- binding 表保存内部对象与外部 provider 的稳定映射关系
- 同步任务表保存异步投递、重试和错误状态
- 外部 provider 可以替换，但内部真源表结构保持稳定

### 7. 数据库按模块归属设计，不按横向层打散

与一期后端“按能力模块封装”的架构保持一致，数据库也建议遵循同样原则：

- 每张核心业务表都尽量归属于单一模块
- 由模块 owner 负责写入、演进和迁移
- 其他模块优先通过 service / repository 边界访问，不直接跨模块写表
- 一期虽然仍是单库单 schema，但表边界应尽量贴近未来的服务边界
- 后续如果拆服务，优先按整组表迁移，而不是从多个模块共享表中拆字段

## 从 Hermes 借鉴什么

Hermes 值得参考的是它的数据边界设计：

- 用 `sessions + messages` 作为核心对话骨架
- 用独立搜索索引支撑历史检索
- 把长期记忆和用户偏好从消息历史中独立出来
- 用 schema version / migration 管理演进
- 保留 session lineage 能力

EchoMind 建议延续这些思路，但改造成更适合服务端的 PostgreSQL 方案。

13. `tool_invocations`
14. `audit_logs`
15. `model_providers`
16. `model_routes`
17. `app_outbox_events`（可选）

## 按模块划分的表归属

建议把数据库表按模块理解，而不是只按“都是数据库表”理解：

- `session/chat` 模块：`sessions`、`messages`、`session_summaries`
- `memory` 模块：`memory_items`、`memory_links`、`memory_provider_bindings`、`memory_sync_jobs`
- `profile` 模块：`users`、`profiles`、`profile_revisions`、`profile_provider_bindings`、`profile_sync_jobs`
- `tools` 模块：`tool_invocations`
- `models` 模块：`model_providers`、`model_routes`
- `platform/shared` 模块：`audit_logs`、`app_outbox_events`

说明：

- 一期仍是单库部署，不需要真的拆库拆 schema。
- 这里强调的是“表归属权”和“写入边界”。
- 例如 `memory_items` 由 `memory` 模块拥有，`chat` 或 `agent` 不应直接拼 SQL 写入该表，而应通过 `MemoryService` 进入。

## 核心实体关系

### 1. `users`

用途：

- 系统内单用户主表

字段建议：

- `id`：用户主键
- `display_name`：用户显示名称
- `avatar_url`：用户头像地址
- `status`：用户状态
- `created_at`：创建时间
- `updated_at`：更新时间

### 2. `sessions`

用途：

- 表示一次会话的生命周期与元数据
- 对应核心对话骨架

字段建议：

- `id`：会话主键
- `user_id`：所属用户主键
- `parent_session_id`：父会话主键，用于分支、重试、续聊
- `source`：会话来源，例如 `chat`、`retry`、`regenerate`、`import`
- `title`：会话标题
- `status`：会话状态
- `active_model_provider`：当前使用的模型提供方
- `active_model_name`：当前使用的模型名称
- `message_count`：会话消息数量
- `tool_call_count`：会话工具调用次数
- `input_tokens`：输入 token 总数
- `output_tokens`：输出 token 总数
- `started_at`：开始时间
- `ended_at`：结束时间
- `last_message_at`：最近一条消息时间
- `created_at`：创建时间
- `updated_at`：更新时间
- `deleted_at`：删除时间，软删除用

### 3. `messages`

用途：

- 存储会话中的每条消息

字段建议：

- `id`：消息主键
- `session_id`：所属会话主键
- `parent_message_id`：父消息主键，用于重试/改写链路
- `role`：消息角色，例如 `system`、`user`、`assistant`、`tool`
- `message_type`：消息类型，例如 `chat`、`tool_call`、`tool_result`、`summary`、`internal`
- `content_text`：纯文本内容
- `content_json`：结构化内容，适合附件、多模态块、复杂消息体
- `tool_name`：工具名称
- `tool_call_id`：工具调用标识
- `tool_calls`：工具调用列表
- `reasoning_text`：推理文本
- `reasoning_details`：推理结构化明细
- `token_count_input`：输入 token 数
- `token_count_output`：输出 token 数
- `sequence_no`：会话内顺序号
- `is_hidden`：是否隐藏，不参与普通展示
- `created_at`：创建时间

### 4. `session_summaries`

用途：

- 存储会话摘要与压缩结果
- 支撑 context compression 与 recall

字段建议：

- `id`：摘要主键
- `session_id`：所属会话主键
- `summary_type`：摘要类型，例如 `session_summary`、`compression_summary`、`recall_summary`
- `summary_text`：摘要文本
- `summary_json`：结构化摘要
- `source_message_id`：摘要来源消息主键
- `version`：摘要版本号
- `created_at`：创建时间
- `updated_at`：更新时间

## 记忆模块设计

建议采用：

- `memory_items` 作为长期记忆真源
- `memory_links` 作为记忆来源映射
- `memory_provider_bindings` 作为外部 provider 映射层
- `memory_sync_jobs` 作为外部 provider 同步状态与重试载体

### 5. `memory_items`

用途：

- 存储系统内部长期记忆条目

字段建议：

- `id`：记忆条目主键
- `user_id`：所属用户主键
- `scope`：记忆作用域，例如 `global`、`session`、`project`、`preference`、`identity`
- `category`：记忆分类，例如 `fact`、`preference`、`habit`、`goal`、`constraint`、`profile_signal`
- `content`：原始记忆内容
- `content_hash`：记忆内容哈希，用于去重
- `normalized_content`：归一化后的记忆内容
- `source_type`：来源类型，例如 `message`、`session_summary`、`profile_refresh`、`manual`、`imported`
- `confidence`：抽取置信度
- `importance`：记忆重要度
- `state`：记忆状态，例如 `active`、`replaced`、`deleted`、`pending_sync`、`sync_failed`
- `version`：版本号
- `last_used_at`：最近使用时间
- `created_at`：创建时间
- `updated_at`：更新时间
- `deleted_at`：删除时间，软删除用

说明：

- `memory_items` 是系统内部记忆真源
- 即使一期只做单用户，也建议显式保留 `user_id`，因为记忆本质上属于用户范围
- 不直接存储外部 provider 特有的主键与结构
- 外部映射关系由 provider binding 表维护
- 后续如果拆成独立 `memory service`，这张表可以直接作为该模块核心主表迁移

### 6. `memory_links`

用途：

- 建立记忆与来源对象的关系

字段建议：

- `id`：关系主键
- `memory_item_id`：记忆条目主键
- `session_id`：来源会话主键
- `message_id`：来源消息主键
- `link_type`：链接类型，例如 `extracted_from`、`confirmed_by`、`used_in_recall`
- `created_at`：创建时间

### 7. `memory_provider_bindings`

用途：

- 保存内部记忆条目与外部 provider 记录的稳定映射关系

字段建议：

- `id`：映射主键
- `memory_item_id`：对应的记忆条目主键
- `provider`：外部提供方，例如 `mem0`
- `external_record_id`：外部记录主键
- `sync_status`：当前同步状态
- `last_synced_at`：最近同步时间
- `last_error`：最近一次同步错误
- `metadata_json`：provider 侧附加信息
- `created_at`：创建时间
- `updated_at`：更新时间

### 8. `memory_sync_jobs`

用途：

- 记录记忆同步到外部 provider 的任务状态

字段建议：

- `id`：任务主键
- `memory_item_id`：对应的记忆条目主键
- `provider`：同步目标提供方，例如 `mem0`
- `operation`：操作类型，例如 `upsert`、`delete`
- `payload_json`：同步负载
- `status`：任务状态，例如 `pending`、`running`、`succeeded`、`failed`、`dead`
- `retry_count`：重试次数
- `max_retry_count`：最大重试次数
- `next_retry_at`：下次重试时间
- `external_record_id`：外部记录主键
- `idempotency_key`：幂等键
- `last_error`：最近一次错误信息
- `created_at`：创建时间
- `updated_at`：更新时间
- `finished_at`：完成时间

## 检索设计

Hermes 使用了 SQLite FTS5；EchoMind 更适合 PostgreSQL FTS。

一期建议：

- 对 `messages.content_text` 建 FTS 索引
- 对 `session_summaries.summary_text` 建 FTS 索引
- 对 `memory_items.content` 建 FTS 索引

### 检索目标

1. session 内历史消息检索
2. 历史 session recall
3. 长期记忆召回

## 索引建议

### `sessions`

- `idx_sessions_user_updated_at (user_id, updated_at desc)`：按用户和更新时间查询会话
- `idx_sessions_parent_session_id (parent_session_id)`：按父会话查询分支链路
- `idx_sessions_last_message_at (last_message_at desc)`：按最近消息时间排序
- `idx_sessions_deleted_at (deleted_at)`：按删除状态过滤

### `messages`

- `idx_messages_session_sequence_no (session_id, sequence_no)`：按会话内顺序读取消息
- `idx_messages_session_created_at (session_id, created_at)`：按会话和时间查询消息
- `idx_messages_tool_call_id (tool_call_id)`：按工具调用标识查询

### `memory_items`

- `idx_memory_items_user_scope_state (user_id, scope, state)`：按用户、作用域和状态查询记忆
- `idx_memory_items_content_hash (content_hash)`：按内容哈希查重
- `idx_memory_items_last_used_at (last_used_at desc)`：按最近使用时间排序

### `memory_provider_bindings`

- `uniq_memory_provider_bindings_item_provider (memory_item_id, provider)`：同一条记忆对同一 provider 只保留一条稳定映射
- `idx_memory_provider_bindings_provider_status (provider, sync_status)`：按 provider 和同步状态筛选

### `memory_sync_jobs`

- `idx_memory_sync_jobs_status_next_retry_at (status, next_retry_at)`：按状态和重试时间取任务
- `uniq_memory_sync_jobs_idempotency_key (idempotency_key)`：幂等去重

### `profiles`

- `uniq_profiles_user_id (user_id)`：一个用户只保留一份当前画像
- `idx_profiles_version (version desc)`：按版本读取最新画像

### `profile_revisions`

- `idx_profile_revisions_profile_version (profile_id, version desc)`：按画像和版本查询历史

### `profile_provider_bindings`

- `uniq_profile_provider_bindings_profile_provider (profile_id, provider)`：同一份画像对同一 provider 只保留一条稳定映射
- `idx_profile_provider_bindings_provider_status (provider, sync_status)`：按 provider 和同步状态筛选

### `profile_sync_jobs`

- `idx_profile_sync_jobs_status_next_retry_at (status, next_retry_at)`：按状态和重试时间取任务
- `uniq_profile_sync_jobs_idempotency_key (idempotency_key)`：幂等去重

### `tool_invocations`

- `idx_tool_invocations_session_created_at (session_id, created_at desc)`：按会话查看工具调用
- `idx_tool_invocations_status (status)`：按状态查询工具调用

### `audit_logs`

- `idx_audit_logs_resource (resource_type, resource_id)`：按资源查询审计记录
- `idx_audit_logs_created_at (created_at desc)`：按时间查询审计记录

## 一期最小落地建议

如果要尽快开工，一期建议优先实现：

1. `users`
2. `sessions`
3. `messages`
4. `memory_items`
5. `memory_provider_bindings`
6. `profiles`
7. `profile_revisions`
8. `profile_provider_bindings`
9. `tool_invocations`
10. `model_providers`
11. `model_routes`
12. `audit_logs`
13. `memory_sync_jobs`
14. `profile_sync_jobs`

这套已经足够支撑：

- `/api/v1/chat`
- `/api/v1/sessions`
- `/api/v1/profile`
- `/api/v1/models`
- 内部记忆写回
- 内部画像刷新
- 外部 provider 异步同步

并且已经具备按模块迁移的基础：

- `memory` 相关表可整体迁移为记忆服务的数据核心
- `profile` 相关表可整体迁移为用户画像服务或用户域数据核心
- `tools` 相关表可迁移为工具执行与审计服务的一部分

## 推荐结论

基于 Hermes 的可借鉴思路，以及 EchoMind 的服务端形态，一期最合适的数据库设计是：

- 用 PostgreSQL 作为主库
- 用 `sessions + messages` 作为对话主骨架
- 用 `memory_items` 作为内建记忆真源
- 用 `profiles + profile_revisions` 作为内建画像真源
- 用 `memory_provider_bindings + profile_provider_bindings` 维护外部映射
- 用 `memory_sync_jobs + profile_sync_jobs` 管理外部同步
- 用 PostgreSQL FTS 支撑 recall 和 session search
- 用 `tool_invocations + audit_logs` 支撑调试和治理

从模块边界角度看，还应额外满足：

- 真源表尽量只服务一个核心模块
- 模块写入边界清晰，跨模块尽量走 service 而不是直接改表
- 表设计优先支持“整模块迁移”，而不是支持“横向分层目录”

这样设计的好处是：

- 一期足够轻，能直接开工
- 不会被第三方记忆或画像服务绑死
- 后续扩展时，内部真源不需要大改
