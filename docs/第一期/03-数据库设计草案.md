# 数据库设计草案

## 目标

为一期后端提供一套**可直接落地、能支撑 MVP** 的数据库设计。

这一版重点解决：

- 用户账号与登录身份绑定
- 聊天会话与消息存储
- 对话运行状态管理与取消控制
- 长期记忆内建存储
- 用户画像内建存储
- 外部 memory / profile provider 同步状态管理
- 会话检索与 FTS
- 工具调用日志
- 审计日志
- 模型配置与当前路由状态

## 设计原则

### 1. 一期做轻量用户体系，不做多租户

一期建议直接纳入最小用户与认证能力：

- 支持账号密码登录
- 登录成功后签发 JWT access token
- 后续请求通过 token 解析 `user_id`
- token 状态由 Redis 管理，不单独设计 refresh token
- 一期按同域部署设计 Cookie 认证约束

但一期仍然不考虑：

- 多租户
- 组织空间
- 复杂权限系统
- 第三方登录

认证补充约束：

- JWT claims 至少包含：`sub(user_id)`、`jti`、`exp`
- access token 通过 `HttpOnly Cookie` 传递
- 一期同域部署下至少校验 `Origin/Referer`，降低 CSRF 风险
- token 状态不写数据库主表，由 Redis 维护

### 2. 内部数据库是真源，外部平台是同步目标

对于长期记忆和用户画像，一期建议采用：

- 内部数据库作为**系统真源**
- Mem0 / Honcho 作为**外部同步目标**

不要把外部平台当作主数据源。

### 3. 结构化主字段入列，灵活数据放 `jsonb`

建议采用：

- 高频筛选、排序、关联字段放独立列
- 可变结构内容放 `jsonb`

例如：

- `messages.role`、`messages.session_id` 放独立列
- `messages.tool_calls`、`messages.reasoning_details` 放 `jsonb`
- `profiles.preferences`、`profiles.work_habits` 放 `jsonb`

派生字段补充约束：

- `sessions.message_count`、`sessions.tool_call_count`、`sessions.input_tokens`、`sessions.output_tokens`、`sessions.last_message_at` 属于派生聚合字段。
- 一期由 `SessionService` 单点维护这些字段，并提供离线重建能力作为数据修复兜底。

### 4. 一期优先 PostgreSQL

参考 Hermes 的设计，它用 SQLite + WAL + FTS5 完成单机持久化，这对于 CLI 型产品很合适。

但 EchoMind 一期是后端服务形态，建议直接采用：

- PostgreSQL 作为主库
- PostgreSQL FTS 作为会话检索能力
- Redis 作为缓存、轻量任务队列和 token 状态存储

### 5. 核心主表保持 provider 无关

对于记忆和画像模块，建议采用：

- 核心业务字段只描述系统内部语义
- 不在核心主表中引入过多第三方平台特有字段
- 第三方平台相关信息放到 binding 表或同步任务表

### 6. 外部平台接入采用 binding 表 + 同步任务

建议采用：

- binding 表保存内部对象与外部 provider 的稳定映射关系
- 同步任务表保存异步投递、重试和错误状态
- 外部 provider 可以替换，但内部真源表结构保持稳定

### 7. 数据库按模块归属设计，不按横向层打散

与一期后端“按能力模块封装”的架构保持一致，数据库也建议遵循同样原则：

- 每张核心业务表都尽量归属于单一模块
- 由模块 owner 负责写入、演进和迁移
- 其他模块优先通过 service / repository 边界访问，不直接跨模块写表
- 一期虽然仍是单库单 schema，但表边界应尽量贴近未来的服务边界
- 后续如果拆服务，优先按整组表迁移，而不是从多个模块共享表中拆字段

## 从 Hermes 借鉴什么

Hermes 值得参考的是它的数据边界设计：

- 用 `sessions + messages` 作为核心对话骨架
- 用独立搜索索引支撑历史检索
- 把长期记忆和用户偏好从消息历史中独立出来
- 用 schema version / migration 管理演进
- 保留会话演进与候选结果扩展能力

EchoMind 建议延续这些思路，但改造成更适合服务端的 PostgreSQL 方案。

## 一期推荐数据库总览

一期建议包含以下表：

1. `users`
2. `sessions`
3. `messages`
4. `session_summaries`
5. `memory_items`
6. `memory_provider_bindings`
7. `memory_sync_jobs`
8. `profiles`
9. `profile_revisions`
10. `profile_provider_bindings`
11. `profile_sync_jobs`
12. `tool_invocations`
13. `audit_logs`
14. `model_providers`
15. `model_routes`
16. `app_outbox_events`（可选，二期增强）

说明：

- 登录态不落数据库主表，token 状态放 Redis 管理。
- 数据库中的 `users` 负责登录身份真源与基础账号字段。
- 一期不提供 session 删除能力，`sessions` 作为对话主骨架长期保留。

## 按模块划分的表归属

建议把数据库表按模块理解，而不是只按“都是数据库表”理解：

- `session` 模块：`sessions`、`messages`、`session_summaries`
- `user` 模块：`users`
- `memory` 模块：`memory_items`、`memory_provider_bindings`、`memory_sync_jobs`
- `profile` 模块：`profiles`、`profile_revisions`、`profile_provider_bindings`、`profile_sync_jobs`
- `tools` 模块：`tool_invocations`
- `models` 模块：`model_providers`、`model_routes`
- `platform/shared` 模块：`audit_logs`、`app_outbox_events`

说明：

- 一期仍是单库部署，不需要真的拆库拆 schema。
- 这里强调的是“表归属权”和“写入边界”。
- 例如 `memory_items` 由 `memory` 模块拥有，`chat` 或 `agent` 不应直接拼 SQL 写入该表，而应通过 `MemoryService` 进入。
- 同理，`sessions/messages/session_summaries` 由 `session` 模块拥有，`chat` 只通过 `SessionService` 进入。

## 核心实体关系

### 1. `users`

用途：

- 系统内用户账号主表
- 作为登录身份真源，与 `profile` 模块分离

字段建议：

- `id`：用户主键
- `username`：登录账号，系统内唯一
- `password_hash`：密码哈希
- `status`：用户状态
- `last_login_at`：最近登录时间
- `created_at`：创建时间
- `updated_at`：更新时间

说明：

- 一期仅做账号密码登录，不单独设计 refresh token 表。
- access token 建议采用 JWT，状态通过 Redis 维护。
- 后续 `sessions`、`memory_items`、`profiles` 等业务数据统一通过 `user_id` 关联当前用户。
- `users` 只承载登录身份与账号凭证，用户名称由 `profiles` 维护。

### 2. `sessions`

用途：

- 表示一次会话的生命周期与元数据
- 对应核心对话骨架

字段建议：

- `id`：会话主键
- `user_id`：所属用户主键
- `title`：会话标题
- `status`：会话状态，例如 `active`、`completed`、`aborted`
- `active_model_provider`：当前使用的模型提供方
- `active_model_name`：当前使用的模型名称
- `message_count`：会话消息数量
- `tool_call_count`：会话工具调用次数
- `input_tokens`：输入 token 总数
- `output_tokens`：输出 token 总数
- `started_at`：开始时间
- `ended_at`：结束时间
- `last_message_at`：最近一条消息时间
- `created_at`：创建时间
- `updated_at`：更新时间

- `retry` 不创建子 session，而是在原 session 内对最近一个 assistant 主结果做物理覆盖
- `regenerate` 不创建子 session，而是在单个 session 内对最近一个 assistant 回复组保留多个候选结果
- 候选结果不出现在 session 列表层，而是在对应 assistant 回复组下查看
- 当用户基于某个候选结果继续对话时，该候选立即转正为主结果；之后该回复组锁定，不再允许切换主结果
- 一期 assistant 回复组与候选结果的核心语义不放在 `sessions` 表，而放在 `messages` 表侧表达
- 同一回复组下“唯一主结果”的更细约束，由 `session` 模块详细设计收口。

### 3. `messages`

用途：

- 存储会话中的每条消息

字段建议：

- `id`：消息主键
- `session_id`：所属会话主键
- `assistant_message_id`：所属 assistant 结果消息主键，用于 tool 消息归属
- `response_group_id`：assistant 回复组归组标识
- `role`：消息角色，例如 `system`、`user`、`assistant`、`tool`
- `message_type`：消息类型，例如 `chat`、`tool_call`、`tool_result`、`summary`、`internal`
- `content_text`：纯文本内容
- `content_json`：结构化内容，适合附件、多模态块、复杂消息体
- `tool_name`：工具名称
- `tool_call_id`：工具调用标识
- `tool_calls`：工具调用列表
- `reasoning_text`：推理文本
- `reasoning_details`：推理结构化明细
- `token_count_input`：输入 token 数
- `token_count_output`：输出 token 数
- `sequence_no`：会话内顺序号
- `is_current_response`：是否为当前主结果
- `response_locked_at`：回复组锁定时间
- `is_hidden`：是否隐藏，不参与普通展示
- `created_at`：创建时间

说明：

- 一期所有真实落库消息都占用 `sequence_no`，包括候选 assistant 消息与其 tool 消息
- `response_group_id` 用于把同一个 assistant 回复组下的主结果、候选结果及其 tool 轨迹归组
- 同一 `response_group_id` 下，`role = 'assistant' and is_current_response = true` 的消息是当前主结果
- 同一 `response_group_id` 下，`role = 'assistant' and is_current_response = false` 的消息是候选结果
- `tool` 消息不参与主结果判定，只通过 `assistant_message_id` 和 `response_group_id` 归属到对应 assistant 结果
- `retry` 直接更新最近一个 assistant 主结果，不新增新的主结果 message
- `regenerate` 为最近一个 assistant 回复组新增候选 message，并保留当前主结果不变
- 当用户基于某个候选结果继续发言时，应用层将该候选切为 `is_current_response = true`，并将该回复组标记锁定

### 4. `session_summaries`

用途：

- 存储会话摘要与压缩结果
- 支撑 context compression 与 recall

字段建议：

- `id`：摘要主键
- `session_id`：所属会话主键
- `summary_type`：摘要类型，例如 `session_summary`、`compression_summary`、`recall_summary`
- `summary_text`：摘要文本
- `summary_json`：结构化摘要
- `source_message_id`：摘要来源消息主键
- `version`：摘要版本号
- `created_at`：创建时间
- `updated_at`：更新时间

## 记忆模块设计

建议采用：

- `memory_items` 作为长期记忆真源
- `memory_provider_bindings` 作为外部 provider 映射层
- `memory_sync_jobs` 作为外部 provider 同步状态与重试载体

### 5. `memory_items`

用途：

- 存储系统内部长期记忆条目

字段建议：

- `id`：记忆条目主键
- `user_id`：所属用户主键
- `scope`：记忆作用域，例如 `global`、`session`、`project`、`preference`、`identity`
- `category`：记忆分类，例如 `fact`、`preference`、`habit`、`goal`、`constraint`、`profile_signal`
- `content`：原始记忆内容
- `content_hash`：记忆内容哈希，用于去重
- `normalized_content`：归一化后的记忆内容
- `source_type`：来源类型，例如 `message`、`session_summary`、`profile_refresh`、`manual`、`imported`
- `confidence`：抽取置信度
- `importance`：记忆重要度
- `state`：记忆状态，例如 `active`、`replaced`、`deleted`、`pending_sync`、`sync_failed`
- `version`：版本号
- `last_used_at`：最近使用时间
- `created_at`：创建时间
- `updated_at`：更新时间
- `deleted_at`：删除时间，软删除用

说明：

- `memory_items` 是系统内部记忆真源
- 一期从一开始就按真实 `user_id` 组织记忆数据，避免后续从“伪单用户”重构
- 记忆去重唯一性不应做全局约束，至少应收敛到 `user_id + scope` 维度
- 同一 `(user_id, scope, content_hash)` 的记忆生命周期内复用原记录，通过 `state/version` 表达替换、删除和恢复，而不是重复插入新行。
- 不直接存储外部 provider 特有的主键与结构
- 外部映射关系由 provider binding 表维护
- 后续如果拆成独立 `memory service`，这张表可以直接作为该模块核心主表迁移

### 6. `memory_provider_bindings`

用途：

- 保存内部记忆条目与外部 provider 记录的稳定映射关系

字段建议：

- `id`：映射主键
- `memory_item_id`：对应的记忆条目主键
- `provider`：外部提供方，例如 `mem0`
- `external_record_id`：外部记录主键
- `sync_status`：当前同步状态
- `last_synced_at`：最近同步时间
- `last_error`：最近一次同步错误
- `metadata_json`：provider 侧附加信息
- `created_at`：创建时间
- `updated_at`：更新时间

### 7. `memory_sync_jobs`

用途：

- 记录记忆同步到外部 provider 的任务状态

字段建议：

- `id`：任务主键
- `memory_item_id`：对应的记忆条目主键
- `provider`：同步目标提供方，例如 `mem0`
- `operation`：操作类型，例如 `upsert`、`delete`
- `payload_json`：同步负载
- `status`：任务状态，例如 `pending`、`running`、`succeeded`、`failed`、`dead`
- `retry_count`：重试次数
- `max_retry_count`：最大重试次数
- `next_retry_at`：下次重试时间
- `external_record_id`：外部记录主键
- `idempotency_key`：幂等键
- `last_error`：最近一次错误信息
- `created_at`：创建时间
- `updated_at`：更新时间
- `finished_at`：完成时间

## 画像模块设计

建议采用：

- `profiles` 保存当前画像
- `profile_revisions` 保存历史版本
- `profile_provider_bindings` 保存外部 provider 映射关系
- `profile_sync_jobs` 保存外部 provider 同步状态

### 8. `profiles`

用途：

- 保存当前生效的用户画像

字段建议：

- `id`：画像主键
- `user_id`：所属用户主键
- `display_name`：用户名称
- `preferences`：偏好信息
- `communication_style`：沟通风格
- `work_habits`：工作习惯
- `domain_context`：领域上下文
- `personality_signals`：人格信号
- `profile_summary`：画像摘要
- `version`：画像版本号
- `last_refreshed_at`：最近刷新时间
- `created_at`：创建时间
- `updated_at`：更新时间

说明：

- `profiles` 是系统内部当前画像真源
- 不直接存储外部 provider 特有的主键与结构
- 外部映射关系由 provider binding 表维护
- 用户名称以 `profiles.display_name` 为真源，避免在 `users` 与 `profiles` 之间双写

### 9. `profile_revisions`

用途：

- 记录画像每次变化的历史快照

字段建议：

- `id`：版本记录主键
- `profile_id`：所属画像主键
- `version`：版本号
- `change_source`：变更来源，例如 `conversation_extract`、`refresh_job`、`imported`
- `snapshot_json`：完整快照
- `diff_json`：增量变化
- `created_at`：创建时间

说明：

- `profile_revisions` 保存系统内部画像演进历史
- 外部 provider 的同步状态不直接放入版本主表
- 画像刷新建议走异步流程；刷新失败不应阻塞聊天主链路，而应进入异步重试或错误记录。
- 一期内部画像刷新任务不单独持久化，只记录刷新后的画像版本、外部同步任务状态和失败审计。

### 10. `profile_provider_bindings`

用途：

- 保存内部画像与外部 provider 记录的稳定映射关系

字段建议：

- `id`：映射主键
- `profile_id`：所属画像主键
- `profile_revision_id`：最近一次同步的画像版本主键
- `provider`：外部提供方，例如 `honcho`
- `external_record_id`：外部记录主键
- `sync_status`：当前同步状态
- `last_synced_at`：最近同步时间
- `last_error`：最近一次同步错误
- `metadata_json`：provider 侧附加信息
- `created_at`：创建时间
- `updated_at`：更新时间

### 11. `profile_sync_jobs`

用途：

- 记录画像同步到外部 provider 的任务状态

字段建议：

- `id`：任务主键
- `profile_revision_id`：对应的画像版本主键
- `provider`：同步目标提供方，例如 `honcho`
- `operation`：操作类型，例如 `upsert`、`delete`
- `payload_json`：同步负载
- `status`：任务状态，例如 `pending`、`running`、`succeeded`、`failed`、`dead`
- `retry_count`：重试次数
- `max_retry_count`：最大重试次数
- `next_retry_at`：下次重试时间
- `external_record_id`：外部记录主键
- `idempotency_key`：幂等键
- `last_error`：最近一次错误信息
- `created_at`：创建时间
- `updated_at`：更新时间
- `finished_at`：完成时间

## 工具、模型与审计

### 12. `tool_invocations`

用途：

- 记录工具调用过程和结果

字段建议：

- `id`：工具调用主键
- `session_id`：所属会话主键
- `message_id`：触发该工具调用的消息主键
- `run_id`：所属运行标识
- `tool_name`：工具名称
- `tool_namespace`：工具命名空间
- `arguments_json`：调用参数
- `result_json`：执行结果
- `status`：调用状态，例如 `pending`、`running`、`succeeded`、`failed`、`cancelled`
- `error_message`：错误信息
- `started_at`：开始时间
- `ended_at`：结束时间
- `created_at`：创建时间

说明：

- 一期工具默认自动执行，不设计人工审批状态字段。
- 即使运行态主要存 Redis，也建议通过 `run_id` 保留数据库内的稳定追踪关系。
- `tool_invocations` 的 `tool_call_id` 去重规则、同一 `run_id` 下的唯一性约束和更细的追踪规则，留到 `tools` 模块详细设计阶段收口。

### 13. `audit_logs`

用途：

- 记录关键业务操作和系统事件

字段建议：

- `id`：审计日志主键
- `request_id`：请求标识
- `actor_type`：操作者类型
- `actor_id`：操作者标识
- `action`：动作名称
- `resource_type`：资源类型
- `resource_id`：资源标识
- `before_json`：变更前数据
- `after_json`：变更后数据
- `metadata_json`：附加信息
- `created_at`：创建时间

## 模型配置

### 14. `model_providers`

用途：

- 保存可用模型配置

字段建议：

- `id`：模型提供方配置主键
- `provider`：提供方名称
- `model_name`：模型名称
- `display_name`：展示名称
- `base_url`：接口地址
- `api_mode`：API 模式
- `capabilities_json`：能力描述

### 15. `model_routes`

用途：

- 保存用户级的实际模型路由状态

字段建议：

- `id`：路由主键
- `user_id`：所属用户主键
- `route_type`：路由类型，例如 `chat_primary`、`summary`、`recall`
- `provider`：当前提供方
- `model_name`：当前模型名称
- `reason`：设置原因
- `is_active`：是否启用
- `created_at`：创建时间
- `updated_at`：更新时间

说明：

- 一期模型切换按用户生效，不做系统全局共享路由。
- 当用户不存在自己的 `model_routes` 时，读取当前模型应回退到全局唯一的 `model_providers.is_default = true` 默认模型。
- 一期不设计系统级 fallback 模型，`model_routes` 只表达用户当前主模型选择。
- `model_routes` 继续保存 `provider + model_name` 文本组合，并由应用层校验它必须存在于平台模型池且处于启用状态。
- 模型禁用、悬空引用和更细的路由闭环规则，留到 `models` 模块详细设计阶段收口。

## Redis 状态设计

一期建议将短生命周期状态统一放 Redis，而不新增数据库主表：

- `auth:token:{jti}`：记录 access token 状态、对应 `user_id`、过期时间
- `chat:run:{run_id}`：记录 chat run 的 `status`、`user_id`、`session_id`、`cancel_requested`

说明：

- `POST /api/v1/chat/stop` 采用软取消语义，即将 `cancel_requested` 设为 `true`
- `agent runtime` 在模型流式输出、tool call 边界和写回前检查该状态
- 这类运行态不属于长期业务真源，一期继续只放 Redis，不单独落数据库主表
- `chat:run:{run_id}` 的 TTL、完成后保留时长和异常清理细则，留到 `chat` 模块详细设计阶段收口。
- 这里的 run 语义与架构文档保持一致：一期只保留 Redis 临时运行态，不设计数据库持久化 run 记录。

初始化补充：

- 一期首个可登录账号建议通过 seed 脚本初始化，而不是开放注册接口。
- 一期不开放人工画像编辑，`profiles` 主要承接自动抽取与刷新结果。
- seed 脚本负责写入首个 `users` 账号记录，供 `/api/v1/auth/login` 使用。

- `profile` 相关表可整体迁移为用户画像服务或用户域数据核心
- `tools` 相关表可迁移为工具执行与审计服务的一部分

## 推荐结论

基于 Hermes 的可借鉴思路，以及 EchoMind 的服务端形态，一期最合适的数据库设计是：

- 用 PostgreSQL 作为主库
- 用 `users` 作为账号与身份真源
- 用 `sessions + messages` 作为对话主骨架
- 用 `memory_items` 作为内建记忆真源
- 用 `profiles + profile_revisions` 作为内建画像真源
- 用 `memory_provider_bindings + profile_provider_bindings` 维护外部映射
- 用 `memory_sync_jobs + profile_sync_jobs` 管理外部同步
- 用 PostgreSQL FTS 支撑 recall 和 session search
- 用 Redis 管理 JWT access token 状态
- 用 `tool_invocations + audit_logs` 支撑自动工具执行的调试和治理

从模块边界角度看，还应额外满足：

- 真源表尽量只服务一个核心模块
- 模块写入边界清晰，跨模块尽量走 service 而不是直接改表
- 表设计优先支持“整模块迁移”，而不是支持“横向分层目录”

这样设计的好处是：

- 一期足够轻，能直接开工
- 不会被第三方记忆或画像服务绑死
- 后续扩展时，内部真源不需要大改
