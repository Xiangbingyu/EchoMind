# 整体架构草案

## 目标

做一个升级版个人助手，具备长期记忆、用户画像、跨会话连续性、多模型接入、工具调用、上下文压缩能力，并支持企业级扩展。

## 结论

核心原则：

- 后端是记忆与画像的唯一真相源。
- 前端只保存临时态、展示态和草稿态。
- 工具调用主协议采用 `function calling`。
- Agent 编排自己掌控，不把核心链路完全托管给 LangChain Agent。
- `LangGraph` 只做编排，不做黑盒 runtime。
- 多模型采用“主模型 + 辅助模型槽位”模式，而不是多主脑协作。

## 对标参考

### OpenCode

- 借鉴点：agent 编排、工具调用、权限控制、上下文压缩、provider 无关接入。
- 不作为主对标记忆系统。

### Hermes Agent

- 借鉴点：长期记忆、用户画像、session search、上下文压缩、prompt caching、辅助模型槽位、后台异步维护。
- 不引入它的 skills 机制作为一期核心。

## 技术选型

- 前端：React / Next.js
- 后端：FastAPI
- 编排：LangGraph
- 模型接入：自研 Provider Router
- 长期记忆：内建轻量记忆 + Mem0
- 用户画像：内建 user profile + Honcho
- 会话检索：SQLite/PostgreSQL FTS + 摘要
- 缓存：以稳定前缀为核心的 prompt caching
- 异步任务：Redis Queue / Arq / Celery 之一，先轻后重

## 总体架构

```text
Front-end (React/Next.js)
  -> FastAPI API Layer
      -> Agent Runtime
          -> Prompt Assembler
          -> Recall Orchestrator
          -> Memory Orchestrator
          -> Context Engine
          -> Tool Dispatcher
          -> Provider Router
          -> Model Call
      -> Session Store
      -> Memory Store
      -> Profile Store
      -> Async Jobs
```

## 模块划分

### 1. 前端层

职责：

- 会话界面
- 记忆/画像展示
- 历史检索展示
- 模型与工具开关
- 审批弹窗

原则：

- 不作为长期记忆真相源。
- 只缓存临时 UI 状态。

### 2. API 层

职责：

- `/chat/send`
- `/sessions/*`
- `/memory/*`
- `/profile/*`
- `/tools/*`
- `/models/*`
- `/admin/*`

### 3. Agent Runtime

职责：

- prompt assembly
- tool calling loop
- 历史召回
- 记忆写入
- 上下文压缩
- provider 路由
- fallback 处理

### 4. Prompt Assembler

稳定前缀建议顺序：

1. agent identity
2. 工具使用规则
3. 内建基础记忆
4. user profile
5. 轻量 Honcho 画像块
6. 必要的系统级约束
7. 项目/租户上下文

临时注入层：

- 当前轮 session recall
- Mem0 recall
- 本轮工具结果摘要
- 当前任务状态
- 压缩生成的中间摘要

### 5. Memory Orchestrator

职责：

- 维护内建轻量记忆
- 调用 Mem0
- 调用 Honcho
- 记忆去重、替换、删除
- 记忆写入策略控制
- 记忆版本化与审计

建议分层：

- `BuiltInMemory`
- `Mem0Adapter`
- `HonchoAdapter`
- `MemoryPolicy`

### 6. Profile Engine

职责：

- 用户身份
- 偏好
- 沟通风格
- 工作习惯
- 性格倾向
- 稳定标签

建议：

- 基础画像存在内部数据库
- Honcho 负责更深层的跨会话建模

### 7. Session Store

职责：

- 全量对话存档
- 会话 lineage
- source tagging
- FTS 检索
- 摘要输入源

建议：

- 一期可用 SQLite + FTS5
- 后续可迁移到 PostgreSQL + FTS + pgvector

### 8. Recall Orchestrator

职责：

- 当前轮是否需要历史召回
- 召回来源排序
- 召回结果压缩
- 回注当前 prompt

召回顺序建议：

1. 内建轻量记忆
2. user profile
3. Mem0 recall
4. Honcho recall
5. Session Search summary

### 9. Context Engine

职责：

- token 估算
- preflight compression
- tool output prune
- head/middle/tail 切分
- structured summary
- iterative summary update

压缩策略：

- 先裁剪旧 tool outputs
- 再压缩中段对话
- 保留头部与尾部
- 摘要支持迭代更新

### 10. Tool Dispatcher

职责：

- function calling 分发
- 工具权限校验
- dangerous action 审批
- 工具结果注入 history

协议建议：

- 主路径：function calling
- 兜底路径：ReAct 文本适配器

## Agent Loop

建议流程：

1. 用户消息入队
2. 构建稳定前缀
3. 读取必要记忆与画像
4. 必要时召回历史会话
5. 判断是否需要压缩
6. 组装 messages
7. 调用主模型
8. 处理 tool calls
9. 工具执行后回写 history
10. 继续循环直到终止
11. 输出最终回复
12. 异步落库和同步外部 provider

## 多模型策略

不做多主脑协作，采用单主模型 + 辅助模型槽位：

- 主模型：对话、决策、工具调用
- 压缩模型：上下文摘要
- 检索摘要模型：session search 结果整合
- 记忆抽取模型：从对话中抽取稳定事实
- 画像增强模型：补充 Honcho 结论

## 异步边界

### 必须同步

- 当前轮 prompt assembly
- 必要 recall
- 工具调用
- preflight compression
- provider routing

### 适合异步

- memory writeback
- Honcho sync
- session indexing
- long summary refresh
- analytics / audit
- profile consolidation

### 半同步

- session search
- Honcho 画像补充
- Mem0 recall/writeback
- 压缩摘要更新

## Session Search 方案

一期建议：

- FTS 搜索为主
- 相邻上下文截取
- LLM 做简要摘要
- 回注当前回合

后续增强：

- embedding rerank
- 时间权重
- task-aware recall
- profile-aware recall

不建议一期直接做全量 embedding-first RAG。

## Prompt Caching 方案

稳定前缀尽量不变：

- agent identity
- 工具规则
- 内建记忆
- user profile
- 稳定画像块
- 项目/租户约束

可变层每轮重算：

- session recall
- 当前工具结果
- 当前任务状态
- 压缩摘要

## 风险点

- 记忆污染
- 画像漂移
- prompt 膨胀
- cache 失效
- 多 provider 冲突
- 工具误操作
- 异步写回延迟导致状态不一致

## MVP 范围

一期只做：

1. 多模型接入
2. 后端会话存储
3. 内建轻量记忆
4. Mem0 长期记忆
5. Honcho 用户画像
6. FTS 版 session search
7. 上下文压缩
8. function calling 工具调用
9. prompt caching 基础策略

## 二期再做

- 更高级的检索增强
- 多租户
- 企业权限与审计
- skills/knowledge automation
- 更强的 profile evolution
- 更完善的异步任务调度

## 最终架构结论

这个项目的核心不是“聊天”，而是“会成长的 assistant runtime”。

最佳路线是：

- function calling 做工具协议
- 自研 agent loop 做控制中枢
- LangGraph 做工作流编排
- 内建轻量记忆 + Mem0 做长期记忆
- 内建 user profile + Honcho 做人物画像
- FTS + summary 做 session search
- 自研 context engine 做上下文压缩
- 稳定前缀 + 临时注入 做 prompt caching
- 同步主链路 + 异步后台任务 做性能与治理
