# 第一期制作目标与方向

## 一、目标

第一期的目标，不是一次性做成完整的 Agent 生态，而是先做出一个**可用、可演示、可继续扩展**的 AgentHub 工作型产品原型。

这一阶段只聚焦**主流工作模式**，即让用户能够围绕项目开发任务，与一个或多个 Agent 在统一的 IM 形态界面中协作，完成从沟通、拆任务、改代码、看 Diff、预览到部署的基本闭环。

## 二、产品定位

AgentHub 第一期定位为：

+ 一个本地优先的开发协作桌面应用
+ 一个以 IM 交互为主的 Agent 工作平台
+ 一个支持单 Agent 对话与多 Agent 协作的工作系统

它不是普通聊天助手，也不是纯自动化脚本平台，而是面向开发场景的“工作中枢”。

## 三、第一期范围

第一期只考虑工作场景，不考虑休闲模式、人格陪伴、自主探索等偏长期成长能力。

本期优先实现以下能力：

+ 单 Agent 工作对话
+ 多 Agent 工作间协作
+ Orchestrator 协调器任务拆解与汇总
+ 多会话并行
+ 通过 `@` 指令进行 Agent 路由
+ Workspace 装载后开展代码编写与任务执行
+ 代码 Diff 展示
+ 网页预览
+ 一键部署
+ Agent 的 skills、tools、plugins 可装配

## 四、核心方向

### 1. 以工作流闭环为核心

第一期最重要的不是做很多炫技能力，而是形成一条完整工作链路：

1. 用户进入某个工作会话
2. 装载 Workspace / Project 上下文
3. 与 Agent 沟通需求或任务
4. Orchestrator 视情况拆解任务
5. Agent 在对应 Sandbox 中执行编码、测试、构建或预览
6. 平台回传 Diff、测试结果、预览链接、部署结果
7. 用户确认并继续下一步工作

### 2. 以 IM 作为统一交互入口

第一期产品形态以桌面端 IM 为主，不再把聊天看作附属界面，而是把它作为整个平台的统一入口。

用户应尽量通过消息完成：

+ 发起任务
+ 指定 Agent
+ 查看执行状态
+ 审阅 Diff
+ 打开预览
+ 触发部署

### 3. 以 Workspace 作为工作边界

第一期引入 Workspace 作为顶层工作空间，用来承载项目、会话、Agent 工作上下文与执行边界。

在一期中，Workspace 重点承担以下职责：

+ 作为代码工作上下文的装载入口
+ 作为 Project / Sandbox / Proposal 的上层容器
+ 作为后续版本扩展多人协作和长期记忆的基础边界

### 4. 以主流工作模式为主，不做复杂自治

第一期的 Agent 主要服务于明确任务执行，不强调自主行动。

因此本期策略是：

+ 强依赖任务以串行为主
+ 明确可拆分任务才交给 Orchestrator 协调
+ 多 Agent 协作主要用于分工和汇总，不追求复杂自治网络

### 5. 以可装配能力支撑扩展

第一期需要为后续扩展留接口，因此 Agent 不应写死为固定能力集合，而应支持装配：

+ skills
+ tools
+ plugins

这样可以在后续继续扩展不同角色 Agent 的差异化能力。

## 五、技术方向

第一期建议采用以下技术路线：

+ 产品形态：桌面端应用 + 本地后端
+ 后端：Python
+ 存储：SQLite
+ Agent 编排：AgentScope
+ 执行与工具对接：ACP
+ 后续可根据需要接入 Git、容器、远程执行环境和更完整的版本管理能力

## 六、本期不优先处理的内容

为了控制复杂度，以下能力不作为第一期重点：

+ 休闲模式
+ Agent 自主探索
+ 强自治自成长系统
+ 复杂多人在线协同
+ 完整企业级权限体系
+ 复杂云端调度和大规模分布式执行

这些能力可以在后续阶段逐步演进，但不应影响第一期落地。

## 七、阶段性成功标准

如果第一期能够稳定实现下面这条最小闭环，就说明方向是正确的：

+ 用户创建或装载 Workspace
+ 在工作会话中与 Agent 沟通任务
+ Agent 或 Orchestrator 能基于 Workspace 执行代码工作
+ 平台能够返回 Diff、预览或部署结果
+ 用户可以继续在会话中推进下一步协作

这意味着系统已经具备了“工作型 AgentHub”的最小产品价值。
