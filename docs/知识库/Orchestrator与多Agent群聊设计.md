## 一、设计目标
这套设计的目标，不是做一个普通的 AI 群聊，而是做一个可以承接真实工程协作的多 Agent 协同系统。

核心目标如下：

+ 将“开放式多角色群聊”和“主 Agent + 内建专用子代理”两种模式结合起来。
+ 用户既可以直接与任意子 Agent 对话、追问、下发任务，也可以通过主 Agent 进行统一规划、拆解和调度。
+ 保留群聊式协作的自然体验，同时保留任务执行、状态跟踪和结果沉淀所需的秩序。
+ 将群聊、任务、Workspace、Sandbox、Proposal、Git 流程打通，形成完整的工程协作闭环。
+ 为长时、多角色、共享历史的群聊设计一套更强的记忆与上下文压缩机制。

## 二、角色定义
### 1. 用户
用户是需求提出者、优先级决定者和最终审批者。

用户可以：

+ 直接 `@` 某个子 Agent 提问或下发任务
+ 直接 `@Orchestrator` 请求任务拆解、调度或汇总
+ 在群聊中补充上下文、调整优先级、暂停任务、取消任务
+ 审阅 Proposal、确认合并、决定是否继续推进下一轮任务

### 2. 主 Agent（Orchestrator）
`Orchestrator` 是群聊中的主 Agent，负责理解用户目标、拆解任务、分派子 Agent、协调执行顺序、跟踪进度并汇总最终结果。

主 Agent 的职责包括：

+ 理解用户整体需求
+ 将大任务拆成可执行子任务
+ 判断任务之间的依赖关系和并行关系
+ 选择最合适的子 Agent
+ 管理任务状态、优先级和冲突
+ 在群聊中持续汇报整体进度
+ 汇总多个子 Agent 的结果并回传给用户

主 Agent 不应该默认承担大量具体编码工作，它更像项目经理和任务调度中心。

### 3. 子 Agent
子 Agent 是具体执行者，负责在特定专业领域内完成任务。

典型子 Agent 可以包括：

+ `CodeAgent`：代码编写、重构、配置修改
+ `DocAgent`：方案文档、说明文档、总结整理
+ `TestAgent`：测试编写、测试执行、结果分析
+ `ReviewAgent`：Diff 审阅、风险提示、合并建议

子 Agent 可以通过两种方式被唤醒：

+ 用户直接 `@`
+ 主 Agent 在任务编排中进行指派

## 三、Plan 与 Orchestrator 的关系
当前更合理的设计，不是把 `Planner` 单独拆成一个独立 Agent，而是把 `Plan` 作为 `Orchestrator` 内部维护的一份结构化状态。

+ 角色定位：
+ `Orchestrator`：同时负责理解目标、生成 Plan、维护 Plan 进度、调度子 Agent、跟踪状态并在必要时做局部重规划。
+ `Plan`：不是独立 Agent，而是 `Orchestrator` 持有的一份执行计划对象，用来记录目标、步骤、依赖、同步/异步属性、当前进度和检查点。
+ 这样设计的原因：
+ 每次任务完成后，`Orchestrator` 都需要立即重新检查当前状态。
+ 检查时往往要顺手更新 Plan 进度、判断下一个任务、决定由谁执行。
+ 如果再引入独立 `Planner`，会额外增加角色同步成本和控制权复杂度。

## 四、Orchestrator 的持续检查与重新分配
`Orchestrator` 不应该只在开始时规划一次，也不应该等所有任务都完成后才重新思考。

+ 默认规则：
+ 每个任务完成后，都触发一次 `Orchestrator` 检查。
+ 每个同步任务完成后，`Orchestrator` 都必须判断下一个同步任务应该由谁执行，再进行重新分配。
+ 每个异步任务完成后，`Orchestrator` 也要检查是否解锁了新的依赖、是否需要补充任务、是否需要汇总阶段结果。
+ 这里要区分三个动作：
+ `check`：任务结束后重新检查当前计划状态。
+ `redispatch`：在现有 `Plan` 不变的前提下，重新决定下一个任务派给哪个 Agent。
+ `replan`：只有原 `Plan` 已不再适用时，才重建后续步骤。

## 五、正常情况下不修改 Plan，只推进和重调度
默认情况下，任务完成后不应该立刻改写整张 `Plan`，而应该优先：

+ 更新当前 `Plan` 进度
+ 判断后续哪个步骤已经可执行
+ 选择最合适的子 Agent 去接下一步
+ 在必要时插入一个补救任务，但仍尽量保持主 `Plan` 主体不变

也就是说，系统日常运行时主要发生的是“重新检查”和“重新分配”，而不是频繁重规划。

## 六、什么时候才进入 Replan
只有在下面这些情况出现时，`Orchestrator` 才进入 `replan`：

+ 用户明确修改目标、顺序或约束
+ 某个失败会影响后续多个步骤，原计划已不再适用
+ 关键依赖失效，例如测试失败后必须先让 `CodeAgent` 修代码
+ 当前检查点之后的大部分步骤都需要改写

进入 `replan` 后，更合理的方式也不是整张计划从头推翻，而是：

+ 保留已经完成且仍然有效的步骤
+ 以最近检查点为起点
+ 重建后面的步骤
+ 再继续执行

## 七、Orchestrator 的内部自触发思考
`Orchestrator` 需要具备内部自触发的再思考能力，但这不是再引入一个新的 `Orchestrator` 去调度自己，而是事件驱动的内部检查机制。

+ 触发条件可以包括：
+ 某个任务完成
+ 某个任务失败
+ 某个任务阻塞
+ 某个依赖满足
+ 用户插入新的控制指令
+ 触发后的默认行为：
+ 先做一次短检查
+ 判断是否只需要 `redispatch`
+ 只有必要时才进入 `replan`

这意味着 `Orchestrator` 不是一个只会在开始时思考一次的静态调度器，而是一个在执行过程中持续检查、持续推进、必要时再规划的主控角色。
