## 草图映射
+ 该草图体现的是一个统一会话入口的左侧导航结构：同一列中同时存在群聊入口、群内 Agent 入口和普通单聊入口，右侧是当前选中会话的主内容区。
+ 群聊入口映射：图中的“群聊1”应被视为群聊主线程入口，用户进入后看到的是群的共享消息流、公开任务协作状态和所有对群有效的决策内容。
+ 群内单聊入口映射：图中的“主Agent”“codeAgent-群聊1”“docsAgent-群聊1”“testAgent-群聊1”“devAgent-群聊1”更适合被定义为“从群聊1派生出来的群内单聊入口”。其中带有“群聊1”标记的 Agent 项，能够更明确传达其从属于该群的语义。
+ 普通单聊入口映射：图中的“codeAgent”“docsAgent”应被视为脱离具体群上下文的普通单聊入口，它们不默认继承“群聊1”的历史，也不直接属于“群聊1”的共享协作链路。
+ 结构分层建议：左侧导航可以在视觉上把“群聊1”及其下属的群内单聊入口归为同一组，再把“codeAgent”“docsAgent”这类普通单聊归为另一组，帮助用户理解“同样都是点开 Agent，但语义并不相同”。
+ 当前选中态建议：当用户点进“codeAgent-群聊1”这类入口时，右侧主区域应明确显示当前为“群内单聊”，并附带“已读取群聊1上下文”“仅支持咨询”“不支持修改任务和代码执行”等状态提示。
+ 群聊返回建议：从群内单聊返回“群聊1”时，应保留明显的主线程入口，使用户能快速回到公开协作界面完成任务调整、结论确认和执行类指令发布。
+ 转发交互建议：既然群内单聊消息默认不自动同步到群聊，那么右侧聊天区应支持多选消息并转发到“群聊1”，让这张草图不只是表达入口结构，也能落到信息回流路径上。
+ 命名建议：为了降低认知负担，群内单聊入口的命名可以统一采用“Agent 名称 + 所属群聊”的格式，例如“codeAgent @ 群聊1”或“codeAgent-群聊1”，而普通单聊则仅保留 Agent 名称。
+ 实现含义总结：这张图最适合表达的不是“所有 Agent 都是同一种聊天入口”，而是“一个统一列表中容纳三种会话对象，分别对应群聊主线程、群内单聊和普通单聊三种不同边界”。

## 概念定义
+ 群聊：面向多人协作和多 Agent 协同的主会话线程，用于承载公开讨论、任务分配、任务调整、执行决策和共享结果沉淀。凡是会影响群协作状态的操作，都应以群聊作为唯一主入口。
+ 单聊：用户与某个 Agent 之间的一对一独立会话，用于脱离群协作场景的个人咨询、方案讨论、日常问答或其他非群上下文任务。单聊默认不继承某个群聊的历史，也不天然属于某个群的共享协作链路。
+ 群内单聊：从某个群聊内部发起的、用户与群内某个 Agent 之间的一对一辅助会话。该 Agent 在群内单聊中可读取所在群聊的消息历史，并基于这些群聊消息继续对话；但群内单聊中新产生的消息默认不进入群聊历史，只有在用户显式转发后才会同步到群聊。

## 核心定位
+ 群聊是主工作流，负责协作、分工、决策和执行入口。
+ 群内单聊是辅助工作流，负责对某个 Agent 进行定向咨询、澄清、追问和状态了解。
+ 普通单聊是独立工作流，负责不依赖特定群上下文的个人对话，不承担群协作中的共享决策职责。

## 边界规则
+ 主线程规则：所有会影响群协作状态的动作，例如任务创建、任务调整、执行要求变更、Agent 分工变化、结果确认和公开同步，都必须在群聊主线程中完成。
+ 咨询模式规则：群内单聊只支持咨询、解释、追问、原因说明、状态说明、方案补充和信息获取，不支持代码编写、不支持直接修改任务、不支持触发执行性动作，也不应直接写入群协作状态。
+ 独立消息流规则：群聊、普通单聊和群内单聊应被视为三种不同消息流。群聊消息不会自动进入单聊，单聊消息也不会自动进入群聊，避免历史污染和信息噪声扩散。
+ 群聊消息读取规则：群内单聊中的 Agent 应可读取所在群聊的历史消息，以便理解需求背景、当前分工、已有结论和项目阶段，并基于这些群聊消息继续回答问题。
+ 单聊消息隔离规则：群内单聊中新产生的消息默认仅用户与对应 Agent 可见，不属于群内其他成员的共享可见内容，也不会自动计入群聊历史；只有在被明确转发回群聊后，相关内容才进入群共享历史。
+ 一致性规则：如果群聊在持续推进，群内单聊中的 Agent 应尽量基于最新群上下文作答，避免出现与主线程脱节的解释或建议。

## 群内单聊设计
+ 入口规则：用户可在群聊中点击任意一个群成员 Agent，进入与该 Agent 的群内单聊；该单聊在产品语义上从属于当前群聊，但在消息层面保持独立。
+ 身份绑定规则：群内单聊中的 Agent 身份应明确标记为“来自某群聊中的某个 Agent”，避免用户误认为这是一个与群协作无关的普通私聊对象。
+ 能力限制规则：群内单聊中，Agent 可以回答“为什么这样做”“当前进展是什么”“这个方案的风险是什么”“群里某条结论是什么意思”等问题，但不能接受“去改代码”“改任务优先级”“切换执行方向”等指令。
+ 拦截提示规则：当用户在群内单聊中尝试发起执行类请求时，系统应明确提示该操作只能在群聊中进行，并引导用户返回群聊主线程完成操作。
+ 只读协作规则：群内单聊中的 Agent 可以读取群聊消息，但群内单聊本身不直接改写群状态，因此它本质上是一个带群聊消息视野的只读咨询通道。

## 信息同步机制
+ 默认隔离规则：群聊消息可被群内单聊中的 Agent 读取，但群内单聊中新产生的消息默认不自动进入群聊，避免群聊被细碎追问淹没，也避免私下讨论未经筛选就变成公共结论。
+ 手动回流规则：若用户希望把群内单聊中的某些内容同步回群聊，应通过多选消息并转发到群聊的方式完成，而不是依赖系统自动同步。
+ 转发生效规则：只有转发进入群聊后的内容，才视为群聊共享历史的一部分，并可被其他成员、其他 Agent 和后续流程共同引用。
+ 转发筛选规则：转发能力应允许用户选择部分消息，而不是要求整个单聊完整同步，以便用户只回流有价值的结论、解释或澄清内容。
+ 来源标记规则：转发到群聊的消息应保留来源信息，例如来自“与某 Agent 的群内单聊”，帮助群成员理解信息出处和语境。

## 为什么这样设计
+ 降噪规则：群聊承担公开协作，容易积累大量执行信息；把针对单个 Agent 的细节追问放到群内单聊中，可以显著降低主线程噪音。
+ 控制规则：把任务调整和执行动作收口到群聊中，可以避免用户在多个入口分别下指令，导致 Agent 目标漂移或协作状态失真。
+ 连贯规则：让群内单聊中的 Agent 直接读取群聊消息，能保证它的回答始终贴合当前项目背景，而不是脱离上下文地重新理解问题。
+ 沉淀规则：通过“手动转发回群聊”而不是“自动同步”，可以把真正重要的信息重新沉淀到主线程里，形成可共享、可追溯的公共历史。

## 产品提示建议
+ 群内单聊页应明确提示当前 Agent 可读取所在群聊的消息，但当前会话仅支持咨询，不支持执行和任务修改。
+ 当用户尝试在群内单聊中要求写代码、改任务或推进执行时，系统应直接给出边界提示，并提供返回群聊的快捷入口。
+ 群聊中应支持把某个 Agent 作为可点击对象，降低从公开协作切换到定向咨询的成本。
+ 群内单聊消息应支持多选、转发到群聊和附带来源说明，确保信息回流过程足够轻量。

## 一句话定义
+ 该机制的核心是：群聊消息对群内单聊中的 Agent 可见，但群内单聊中新产生的消息默认不进入群聊历史；允许定向咨询，不允许绕开群聊执行；所有真正影响协作状态的动作，统一收口在群聊主线程中。
